## 这篇文章在讲什么

存储器层次结构要解决的核心矛盾，是“大容量”和“高速度”很难同时做到。  
这一章可以始终围绕三个问题去理解：

1. 为什么 Cache 能起作用？
2. 数据在 Cache 里是怎么放、怎么找的？
3. 命中、缺失和写策略会怎样影响性能？

## 局部性原理

局部性原理不仅适用于我们在图书馆查资料的方式，也适用于程序的执行过程。它说明：在任意一小段时间内，程序通常只会访问地址空间中相对较小的一部分内容。

程序的局部性主要体现在两方面：

- **时间局部性**：最近访问过的数据，很可能在不久之后再次被访问。
- **空间局部性**：与最近访问地址相邻的数据，也很可能很快被访问。

存储器层次结构正是利用了这两个性质：

- 利用时间局部性，把最近访问的数据保存在更靠近处理器的上层存储器中。
- 利用空间局部性，把一整块相邻的数据一起搬到上层存储器，而不是只搬一个字。

## 存储器层次结构

在存储器层次结构中，离处理器越近的层次，容量通常越小，但速度越快；离处理器越远的层次，容量越大，但速度越慢。

因此，理想状态是：

- 数据尽量在高层命中，这样访问速度快；
- 即使需要访问低层，也能靠较高的命中率，把整体平均访问时间压低。

### 常见存储器技术

| 技术 | 成本 | 速度 | 常见用途 |
| ---- | ---- | ---- | -------- |
| SRAM | 高 | 最快 | 靠近处理器的 Cache |
| DRAM | 中 | 较快 | 主存储器 |
| 闪存 | 较低 | 较慢 | 手机、SSD 等二级存储 |
| 磁盘 | 最低 | 最慢 | 大容量长期存储 |

## Cache 的核心问题

可以把 Cache 理解成处理器和主存之间的一层“小而快”的缓冲区。  
它依赖局部性原理，尽量在更高层次中找到需要的数据；如果预测失败，再去更低一层把正确数据取回来。

理解 Cache 时，最重要的是回答两个问题：

1. 如何判断一个数据项是否在 Cache 中？
2. 如果它在 Cache 中，如何准确找到它？

#### 直接映射 Cache

###### 什么是直接映射

在直接映射（direct mapped）Cache 中，每个主存块只能映射到 Cache 中唯一的一个位置。

常见的映射方式是：

$$
\text{Cache 行号} = \text{块地址} \bmod \text{Cache 中的块数}
$$

这是一个使用具体数值的例子，能帮你更直观地理解地址是如何被“切分”并映射到缓存中的。

假设我们有一个非常简单的系统：

- **内存地址长度**：16 位（可以寻址 $2^{16}$ 个字节）。
- **缓存大小**：总共 8 个缓存槽（Cache Lines）。
- **块大小（Block Size）**：每个块 4 字节。

------

###### 第一步：计算各部分的位数

在硬件电路中，地址会被分成三个字段：

1. **块偏移（Offset）**：块大小是 4 字节（$2^2$），所以需要 **2 位**来定位块内的具体字节。
2. **索引（Index）**：缓存有 8 个槽（$2^3$），所以需要 **3 位**来决定去哪个槽。
3. **标记（Tag）**：剩下的位数都给标记。$16 - 2 - 3 =$ **11 位**。

------

###### 第二步：具体的地址映射例子

假设 CPU 现在想要访问内存地址：**`0x045E`**。

1. **转换为二进制**：

   `0x045E` = `0000 0100 0101 1110`

2. **切割地址**：

   根据第一步算出的位数，我们从右往左切：

   - **低 2 位 (Offset)**：`10`（十进制的 2，表示要找块里的第 3 个字节）。
   - **中间 3 位 (Index)**：`111`（二进制的 111 是 **7**）。
   - **高 11 位 (Tag)**：`0000 0100 010`。

3. **查找缓存**：

   - **去哪找？** CPU 直接定位到缓存的 **第 7 号槽（Line 7）**。
   - **比对：** 查看第 7 号槽里的“标记”是不是 `0000 0100 010`。
   - **结果：** * 如果标记匹配且有效位（Valid bit）为 1，就是 **Cache Hit（命中）**，直接拿走数据。
     - 如果不匹配，就是 **Cache Miss（缺失）**，需要去内存里把这个块搬进第 7 号槽，并更新标记。

------

###### 第三步：为什么叫“直接映射”？

想象另一个地址：**`0x047E`**。

- 它的二进制是：`0000 0100 0111 1110`
- 它的索引位（Index）同样是 **`111`**。

这意味着地址 `0x045E` 和 `0x047E` **必须争夺同一个位置**（第 7 号槽）。只要其中一个进来，另一个就必须被踢出去。这就是直接映射的局限性——即使缓存其他地方是空的，这两个地址也只能挤在第 7 号槽。

------

###### 总结

你可以把这个过程想象成**查宿舍号**：

- **Index（索引）**：是你的房间号（每个学生根据学号后几位被固定分配到一个房间）。
- **Tag（标记）**：是门牌上的名字（用来确认房间里住的是不是你要找的那个人，因为不同系的学生可能被分到同一个房间号）。
- **Offset（偏移）**：是房间里的床位号。



> 也就是说，相同的低位地址可能对应主存中的不同块，它们会竞争同一个缓存位置。

### 标记位和有效位

既然多个主存块可能映射到同一个 Cache 行，我们就必须在 Cache 中额外保存一部分地址信息，这就是 **Tag（标记位）**。

它的作用是：

- 用索引先定位到某一行；
- 再用 Tag 判断这一行里保存的，是否真的是当前请求的数据。

除了 Tag，还需要一个 **Valid bit（有效位）**：

- `1` 表示这一行目前存的是有效数据；
- `0` 表示这一行还没有被正确填充，或者当前内容不可用。

> 例如刚开机时，Cache 里的内容通常都视为无效，因此有效位会被初始化为 `0`。

### 读操作是怎么发生的

![Cache 读操作示意图](./%E8%AE%A1%E7%AE%97%E6%9C%BA%E7%BB%84%E6%88%90%E5%8E%9F%E7%90%86-%E5%AD%98%E5%82%A8%E5%99%A8%E5%B1%82%E6%AC%A1%E7%BB%93%E6%9E%84.assets/image-20260418151527080.png)

例如读取地址 `00011` 时，可以这样理解：

1. 先根据低位 `011` 找到对应的 Cache 行。
2. 再拿高位 `00` 去和这一行保存的 Tag 比较。
3. 如果 Tag 匹配且有效位为 `1`，说明命中。
4. 如果不匹配，说明缺失，需要从主存中把对应块取回来。

## 直接映射 Cache 的容量计算

下面这一部分的重点是：**Cache 不只是存数据，还要存 Tag 和有效位，因此实际硬件位数会比“名义容量”更大。**

![直接映射 Cache 位数计算示意图](./%E8%AE%A1%E7%AE%97%E6%9C%BA%E7%BB%84%E6%88%90%E5%8E%9F%E7%90%86-%E5%AD%98%E5%82%A8%E5%99%A8%E5%B1%82%E6%AC%A1%E7%BB%93%E6%9E%84.assets/image-20260418152529336.png)

先做几个设定：

- 地址长度为 32 位。
- Cache 一共有 $2^n$ 行，因此索引需要 $n$ 位。
- 每个块有 $2^m$ 个字。
- 每个字为 4 字节，也就是 32 位。

### 地址如何拆分

一个 32 位地址可以拆成三部分：

- **Tag**：$32 - (n + m + 2)$ 位
- **Index**：$n$ 位
- **Offset**：$m + 2$ 位

其中：

- `n` 位用于选中哪一行 Cache；
- `m` 位用于确定块中的哪个字；
- 额外的 `2` 位用于确定一个字中的哪个字节。

### 每一行有多少位

每个 Cache 行包含三部分信息：

- **数据部分**：$2^m$ 个字，每个字 32 位，所以一共是 $2^m \times 32$ 位
- **Tag**：$32 - n - m - 2$ 位
- **Valid bit**：1 位

因此每行总位数为：

$$
2^m \times 32 + (32 - n - m - 2) + 1
= 2^m \times 32 + 31 - n - m
$$

整个 Cache 一共有 $2^n$ 行，所以总位数为：

$$
2^n \times \left(2^m \times 32 + 31 - n - m\right)
$$

### 一张表记住

| 组成部分 | 位数 | 说明 |
| ---- | ---- | ---- |
| 数据 | $2^m \times 32$ | 实际存储的数据内容 |
| Tag | $32 - n - m - 2$ | 用于匹配主存地址 |
| Valid | $1$ | 标记这一行是否有效 |
| 每行总位数 | $2^m \times 32 + 31 - n - m$ | 数据位加控制位 |
| 整个 Cache 总位数 | $2^n \times (2^m \times 32 + 31 - n - m)$ | 硬件实际开销 |
| Cache 命名容量 | $2^n \times 2^m \times 4$ 字节 | 只统计数据部分 |

## Cache 缺失时怎么处理

如果访问的数据不在 Cache 中，就会发生 **Cache miss（缺失）**。  
这时控制单元必须从主存，或者更低一级的 Cache 中把需要的数据取回来，再继续执行。

以指令 Cache 缺失为例，基本流程可以概括为：

1. 把程序计数器 `PC` 的原始值（通常是当前指令地址）送到存储器。
2. 通知主存执行读操作，并等待读完成。
3. 把取回的数据写入 Cache，对应写入 Tag，并设置有效位。
4. 重新取指，这一次目标指令就能在 Cache 中命中。

## 写操作的三种常见策略

写操作比读操作更麻烦，因为如果只修改 Cache 而不修改主存，两者的内容就会不一致。

### 写直达

写直达（write-through）指的是：**每次写 Cache 时，同时也写主存**。

它的优点是实现简单、一致性更容易保证；缺点是每次写都可能拖慢处理器，因为主存写入速度远慢于 Cache。

### 写缓冲

为了减轻写直达带来的等待，可以引入 **写缓冲区（write buffer）**：

- 处理器先把数据写入 Cache 和写缓冲区；
- 后台再由写缓冲区慢慢把数据写回主存。

这样做的好处是：处理器不必每次都同步等待主存完成写入。  
但如果写缓冲区满了，处理器仍然会被迫停下来等待。

### 写回

写回（write-back）指的是：**写操作先只更新 Cache，不立刻更新主存**。

只有当这个 Cache 块将来被替换出去时，才把它写回到更低层存储器。  
这种方式通常效率更高，但实现上会更复杂，实际硬件中通常还需要配合脏位（dirty bit）来判断该块是否被修改过。

## Cache 性能怎么评估

### CPU 时间公式

CPU 时间可以看成两部分之和：

- CPU 真正执行指令消耗的时间
- 因为等待存储系统而阻塞的时间

因此有：

$$
\text{CPU 时间} =
(\text{CPU 执行时钟周期数} + \text{存储器阻塞时钟周期数})
\times \text{时钟周期}
$$

而存储器阻塞周期，又可以拆成读操作阻塞和写操作阻塞。

### 读操作阻塞

$$
\text{读操作阻塞时钟周期数}
=
\frac{\text{读次数}}{\text{程序执行次数}}
\times
\text{读缺失率}
\times
\text{读缺失代价}
$$

### 写操作阻塞

写操作的阻塞通常来自两部分：

1. 写缺失，需要先把数据块取回来；
2. 写缓冲区阻塞，说明写入请求堆积过多。

因此可以写成：

$$
\text{写操作阻塞时钟周期数}
=
\left(
\frac{\text{写次数}}{\text{程序执行次数}}
\times
\text{写缺失率}
\times
\text{写缺失代价}
\right)
+
\text{写缓冲区阻塞}
$$

写缓冲区阻塞往往很难用一个简单公式准确表示，因为它不仅取决于写入频率，还取决于写操作出现的时机。  
如果写缓冲区足够深，而且主存接收写入的速度明显高于程序平均写频率，那么这部分阻塞通常可以近似忽略。

### 平均存储器访问时间

分析 Cache 性能时，一个非常常用的指标是 **AMAT（Average Memory Access Time，平均存储器访问时间）**：

$$
\text{AMAT} = \text{命中时间} + \text{缺失率} \times \text{缺失代价}
$$

这个公式很好记：

- 命中时，花的是命中时间；
- 缺失时，还要额外承担一次 miss penalty（缺失代价）。

## 更灵活的块放置方式

前面讲的直接映射虽然简单，但冲突也比较明显。为了减少**多个块争用同一位置的问题**，还可以使用更灵活的放置方式。

### 全相联

全相联（fully associative）允许一个主存块被放到 Cache 的**任意位置**。

优点是冲突少；缺点是查找时需要比较所有可能位置，实现成本高。

### 组相联

组相联（set associative）介于直接映射和全相联之间。

它的思路是：

1. 先把一个块映射到某个组；
2. 再在这个组内部检查所有行，看哪一行匹配。

> 可以把它理解成“先缩小范围，再在组内并行比较”。

![组相联示意图](./%E8%AE%A1%E7%AE%97%E6%9C%BA%E7%BB%84%E6%88%90%E5%8E%9F%E7%90%86-%E5%AD%98%E5%82%A8%E5%99%A8%E5%B1%82%E6%AC%A1%E7%BB%93%E6%9E%84.assets/image-20260418161324538.png)

在组相联或直接映射 Cache 中，地址仍然可以拆成三部分：

- **索引位**：用来选择某个组
- **Tag**：用来在组内比较、确认是否命中
- **块内偏移**：用来确定块中的具体数据位置

## 替换块时怎么选

当 Cache 缺失时，如果目标位置已经被占满，就必须选择一个已有块替换出去。

- 对于直接映射 Cache，可选位置只有一个，所以不需要“挑”。
- 对于全相联 Cache，所有块都可能成为替换对象。
- 对于组相联 Cache，只需要在被选中的组内部决定替换哪一块。

### LRU：最近最少使用

一种经典的替换策略是 **LRU（Least Recently Used）**，也就是：

> 把最近最少使用的那一块替换出去。

它背后的直觉也来自局部性原理：最近常用的数据，未来继续被访问的概率通常更高。

## 总结

这一章最值得记住的其实就三点：

1. **局部性原理**决定了存储器层次结构为什么有效。
2. **映射方式、Tag 和有效位**决定了数据在 Cache 中怎么放、怎么找。
3. **缺失代价、写策略、相联度和替换策略**共同决定了 Cache 的实际性能。

如果后面继续复习这一章，我觉得最值得单独展开的内容是两块：

- 直接映射、组相联、全相联三种结构的对比题；
- AMAT、缺失率和写策略相关的计算题。

##### 使用多级 cache 结构减少缺失代价

为了进一步减小现代处理器高时钟频率与日益增长的 DRAM 访问时间之间的差距，大多数微处理器都会增加额外一级 cache 。这种二级 cache 通常位于芯片内，当一级 cache 缺失时就会访间二级 cache 。如果二级 cache 中包含所需要的数据，那么一级 cache 的缺失代价就是二级 cache 的访问时间，这要比访问主存快得多。如果一级和二级 cache 中均不包含所需的数据，就需要访存，这样就会产生更大的缺失代价。

一级 cache 和二级 cache 的设计思想明显不同，这是因为对于单级 cache, 其他 cache 的存在改变了最佳选择。特别是两级 cache 的结构中，一级 cache 致力于减少命中时间以获得较短的时钟周期或者较少的流水级，二级 cache 则侧重千改善缺失率以减少长时间的访存代价。
